AI for Engineering Automation

Where AI helps in engineering workflows, how to pick a model, and patterns I use for automation.

This guide is opinionated and practical: it reflects how I use AI as an automation engineer—not a catalog of every tool on the market.

Treat every model as a probabilistic component: it can speed you up or quietly ship subtle mistakes. The sections below are the decision points I run through before leaning on AI for a task.

Where to implement AI (and where not to)

Good candidates share a few signals: repetition, clear inputs, tolerance for a first draft, and a human or automated check before anything reaches production.

  • Boilerplate and scaffolding: test data, config stubs, migration sketches, repetitive CRUD—then you review and harden.
  • Explaining and decomposing: turning logs, stack traces, or legacy scripts into structured notes or a test plan.
  • Refactoring with guardrails: “same behavior, smaller functions” works when tests or typecheckers catch regressions.

Poor fits: anything that needs a guaranteed correct answer with no verification (auth, billing, security policies), or work where the cost of a wrong word is higher than the cost of doing it slowly yourself.

If you cannot name what would falsify a bad answer (a test, a diff, a policy check), pause before outsourcing the thinking.

Distinguishing models and assistants

Capabilities overlap, but I reach for different products for different cognitive loads. Names and UIs change; the split below is about how I use them today.

None of these replace domain judgment—they change how fast you draft, search, and stress-test ideas.

AI tools compared for engineering use
Tool I use it for Watch for
ChatGPT General explanations, rewriting, quick scripts, brainstorming product copy, turning rough notes into checklists. Overconfidence on APIs you did not paste; vague “best practices” without your stack context.
Claude Longer code review–style reads, comparing diffs, algorithmic reasoning, structured refactors across many files when I paste enough context. Still hallucinates imports and edge cases—verify with run/test.
Copilot / Cursor In-IDE completion, small localized edits, boilerplate from surrounding symbols—fastest when the editor already has the repo open. Tunnel vision: it optimizes the current file, not the system.
Perplexity (and similar) Grounded-ish lookup when I want citations to docs or release notes, not when I need executable code without checking. Stale or wrong sources; always open the primary link.

When a task spans “explain the approach” and “edit the repo,” I often start in Claude or ChatGPT for structure, then finish in the IDE with Copilot/Cursor for local syntax and repetition.

Automation patterns that survive review

These are the shapes that have held up on my teams: they assume CI, linters, or a second pair of eyes exist somewhere in the loop.

  1. Spec → generate → verify: paste constraints and interfaces; generate a script or test; run it in a sandbox before merging.
  2. Devil’s advocate: ask explicitly for failure modes, race conditions, and “what would break this in CI?” on a PR you already wrote.
  3. Documentation from truth: generate README or runbook sections from the actual commands in your pipeline file—never the other way around.
  4. Incremental prompts: one coherent step per message (repro command, expected vs actual, hypothesis) beats a wall of context.

The goal is not maximum tokens; it is minimum surprise when someone else runs your automation six months later.

Prompt skeletons

Use plain structure; models mirror the bullets you give them.

Automation / script generation

Goal: (one sentence)
Environment: (OS, runtime, versions)
Inputs / outputs: (files, APIs, env vars)
Constraints: (no network, idempotent, must run in CI)
Deliverable: (language) script + how to run + exit codes

PR / design critique

Change summary: (what you did)
Risk focus: (data integrity, concurrency, security)
Question: List the top 5 ways this could fail in production and how to test for each.
