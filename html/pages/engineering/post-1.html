How Living With MS Changed the Way I Engineer Software

I'm a senior automation engineer living with multiple sclerosis.

MS introduces variability in energy, cognition, and physical strength and consistency. Some days are sharp, where I feel like I can solve any hard technical problem. Some require pacing because my brain decided 'nope'.

That variability changed how I think about engineering systems. I no longer design for ideal conditions. I design for constraints and the "unexpected".

And that has fundamentally improved the way I build software.

Variability Exposes Systems

Recently, I ran into a frustrating issue with my CI pipeline on one of my repos.

Two GitHub Actions jobs were running Playwright tests simultaneously. Locally, everything passed. In CI, tests started failing intermittently.

At first glance, it looked like selector flakiness.

Plot twist – it wasn't.

The real issue was resource contention — concurrent jobs binding to the same ports, spinning up browsers simultaneously, creating race conditions that only appeared under shared runner constraints.

The problem wasn't "fix the selector," it was:

Design the system such that it behaves predictably under constrained resources.

Living with MS makes you sensitive to variability. You stop assuming stability and start engineering for degradation.

That mindset made the difference in debugging the pipeline.

Determinism Is Not a Luxury

When cognitive energy fluctuates, unpredictability becomes expensive.

Non-deterministic pipelines, unclear error messages, hidden state transitions — they all increase cognitive tax.

So I have a strong bias toward determinism.

In my Playwright tests, I ensured:

  • Explicit waits instead of implicit timing assumptions.
  • Isolated servers.
  • Clear teardown logic.
  • Reduced parallelism when resource contention matters.

In CI:

  • Separate concerns across PRs.
  • Controlled concurrency.
  • Clear failure diagnostics.
Deterministic systems reduce mental overhead. They preserve energy — for both humans and machines.

Automation as Energy Conservation

Automation is often framed as productive, fast responses towards repetitive tasks.

For me, it's sustainability.

When I built my personal site, I didn't want manual content maintenance. So I automated it.

Example: GitHub → Website Sync

Instead of manually updating my projects page:

  • A GitHub Action queries the GitHub API.
  • It extracts repository metadata.
  • It transforms that data.
  • It updates the site.
  • It redeploys.

That system:

  • Reduces repetitive effort.
  • Eliminates human sync errors.
  • Keeps the portfolio accurate automatically.

Automation protects against inconsistency — especially when energy fluctuates or brain fog hits.

Building My Own Markup Language

At one point, I was journaling directly into my site.

But writing raw HTML was time consuming and cumbersome.

So I created a lightweight markup system tailored to how I write notes.

I defined:

  • Simple structural markers.
  • Formatting shortcuts.
  • A predictable syntax aligned with my thinking patterns.

Then I built a script to:

  • Parse the custom markup.
  • Transform it into valid HTML.
  • Auto-generate pages.

Why? Because reducing friction, steps, and simplifying processes matters.

When cognitive load increases, even small formatting decisions become noise. Designing a system aligned with my thought process removed unnecessary mental tax.

That's not just accessibility — it's being proactive.

Accessibility Is Architectural

Accessibility is often treated as:

  • A front-end audit.
  • A compliance task.
  • A checklist item.

But accessibility is a system property.

When I build:

  • Clear logs.
  • Idempotent APIs.
  • Retry-safe workflows.
  • Graceful degradation.
  • Offline-capable designs.
That's accessibility. A system that handles degraded network conditions is accessible. A workflow that survives interruption is accessible. An API that tolerates retries is accessible.

Living with neurological variability makes this obvious.

Resilient systems assume variance.

Engineering for Recovery

Constraint changes your default question.

Instead of: "Does this work?"

You ask: "What happens when it partially works?"

Instead of: "Is this correct?"

You ask: "Is this recoverable? What happens if it fails mid-flight?"

In my automation scripts and pipelines, I build for recovery:

  • Fallback logic.
  • Clear error states.
  • Safe retries.
  • Observable behavior.

Because failure isn't rare. It's normal.

And recoverability defines robustness.

Constraint Sharpens Design

Living with MS hasn't reduced my engineering capability.

It refined it.

It sharpened my sensitivity to:

  • Hidden complexity.
  • Cognitive overhead.
  • Fragile state management.
  • Unnecessary manual steps.
  • Silent failure modes.

Constraint removes excess.

It forces clarity.

It rewards simplicity.

Closing

Engineering from constraint produces stronger systems.

It prioritizes:

  • Determinism over assumption.
  • Recovery over perfection.
  • Automation over repetition.
  • Architecture over patchwork.

MS introduced variability into my life.

So I build software that assumes variability too.

And systems that assume variance are more resilient — for everyone.
