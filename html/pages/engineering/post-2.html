Accessibility Is Not Just a Feature

When most teams talk about accessibility, it's usually an afterthought in the process.

It shows up as "extra credit":

  • Add alt text
  • Fix color contrast
  • Ensure keyboard navigation works
  • Run a WCAG audit before release

These are all important because they frame accessibility as a surface-level feature — something that sits on top of the product.

Living with multiple sclerosis and working as an automation engineer, I've come to believe something different:

Accessibility isn't just a feature. It's something to be decided as infrastructure.

And when it is seen that way, the entire way system design is approached, changes.

The Problem With Treating Accessibility Like a Feature

During planning, Feature thinking assumes a "normal" user.

That's when accessibility becomes an exception to that normal case.

You build the system for the majority, and then you patch the edges. But real systems don't behave that way.

People use software:

  • while tired
  • while distracted
  • on poor internet connections
  • on small screens
  • with temporary or permanent disabilities
  • in environments that were never imagined

When systems fail under those conditions, the issue isn't accessibility — the issue is fragility.

Accessibility exposes that fragility.

Accessibility Is Really About Resilience

When engineers talk about infrastructure, we talk about resilience:

  • systems that recover from failure
  • systems that degrade gracefully
  • systems that continue working under constraint or pressure

Accessibility is the same principle applied to users. A resilient system assumes:

  • input might be unexpected
  • networks might be unreliable
  • attention might be limited
  • physical ability might vary

If a product only works under perfect conditions, it isn't robust. It's just optimized for the ideal scenario.

The Connection to Cognitive Load

Living with MS means dealing with cognitive fatigue. Some days your brain processes information slower than usual.

Complex interfaces that might feel manageable to others suddenly become overwhelming.

This changes how you evaluate software.

You start noticing things like:

  • deeply nested menus or convoluted and confusing UIs
  • inconsistent interaction patterns
  • error messages that provide no clear action and leave you wondering "what?"
  • systems that force you to remember steps instead of guiding you

These aren't just UX annoyances, they are an increase in cognitive load and infrastructure.

When software requires excessive mental processes, it excludes users long before a screen reader ever even enters the picture.

Engineering Systems With Accessibility in Mind

Thinking of accessibility as infrastructure changes the engineering priorities. You begin designing for constraint first, not just an afterthought.

For example:

Clear State Machines

Systems should have predictable and clear transitions.

Users should always know:

  • where they are
  • what the system is doing
  • what happens next

Ambiguous system states create confusion, distraction and increase cognitive load.

Deterministic Error Handling

Error messages should answer four questions immediately:

  • What happened?
  • How did it happen?
  • Why did it happen?
  • What can I do about it?

Non-deterministic messages like "Something went wrong" force users to guess what went wrong.

Offline and Intermittent Modes

A reliable system doesn't assume perfect connectivity or ideal scenarios. Basements, elevators and older buildings often have poor signal.

If a task completely fails because the network dropped, that system isn't accessible in the real world and a fallback is necessary. Graceful degradation, indeed, matters.

Automation and Consistency

Automation reduces the number of decisions users need to make by automating tasks to increase efficiency.

Predictable patterns reduce cognitive load.

Consistency isn't just a design preference — it's also infrastructure.

Accessibility Benefits Everyone

The interesting thing about infrastructure improvements towards accessibility is that they help everyone.

The same design choices that help someone with a disability also help:

  • a tired developer using the tool at 2 AM
  • someone working on a slow laptop
  • a user with a bad internet connection
  • a person learning the system for the first time

Accessibility improvements are rarely niche.

They're usually good engineering.

Accessibility as an Engineering Mindset

When accessibility is treated as a feature, teams tend to address it at the end of development or if there is time left in the sprint.

When accessibility is treated as infrastructure, it becomes a must-have from the beginning, not a nice-to-have.

You ask different questions:

  • How does this behave when conditions aren't ideal?
  • What happens when the user is tired or distracted?
  • Does the system guide the user, or does it make them think harder?
  • What if the user doesn't use the system the way it's designed to?

The goal isn't just compliance. The goal is robustness.

Robust systems work for more people, in more environments, under more conditions. That's good engineering.
